# Roadmap Design Skill Blueprint

## 1. 目的

Roadmap Design Skill は、曖昧な課題候補をそのまま「提案文」に変換するのではなく、実装と検証に移れる計画単位へ落とすことを目的とする。

正規出力は機械可読な JSON とし、最低限以下を返す。

- 再定義された課題
- 成功条件
- 仮説
- 解決方針候補
- 実験計画
- フェーズ化されたロードマップ
- リスク
- フォールバック
- 直近の次アクション

## 2. 解くこと / 解かないこと

### 解くこと

- 問題を扱える境界に絞り込む
- 仮説と実験を対応付ける
- 実装順と検証順を持つロードマップへ分解する
- 計画不能時も、何が不足しているかを返す

### 解かないこと

- 課題発見そのもの
- 検索や情報収集
- 実装の自動生成
- 実験の自動実行
- 外部ストレージへの永続化
- マルチテナントや認証

## 3. 今回補完した設計判断

既存ドラフトには価値の核はあった一方で、実装時に迷いやすい点が残っていたため、以下を明示的に固定した。

### 3.1 1 リクエスト 1 課題

MVP では `problem_statement` は 1 件のみ受ける。複数課題の束ね処理は別レイヤーで行う。

### 3.2 同期 MVP

既存 I/F ドラフトには `GET /v1/runs/{run_id}` などの run 参照系があったが、永続化対象外という要件と矛盾していた。MVP は同期 `run` / `validate` のみを正式化し、非同期実行は将来拡張へ切り出した。

### 3.3 契約バージョニング

request / response / error のすべてに `schema_version` を持たせ、後方互換性判断の起点を固定した。初版は `1.0.0` とする。

### 3.4 Traceability

仮説、問題再定義、解決方針は、入力の insight / constraint / evidence を参照できる形にした。計画の根拠を失わないことを優先する。

### 3.5 構造化された次アクション

`next_actions` は文字列配列ではなく、`action_id` `title` `description` `deliverable` `depends_on` を持つオブジェクト配列へ固定した。これにより Task Seed や Issue 化へ直結できる。

### 3.6 出力の安定性

同一入力に対して、配列順序と ID prefix が極端に揺れないことを要求へ追加した。品質差よりも、比較可能性とレビュー容易性を優先する。

## 4. アーキテクチャ方針

- `roadmap_core`: バリデーション後の正規化入力を受け、構造化結果を返す純粋ロジック層
- `wrappers/cli`: ファイル I/O と exit code を担当
- `wrappers/http`: JSON over HTTP の薄い面
- `wrappers/mcp`: tool 名と payload 中継を担当
- `schemas/`: 契約の正本
- `examples/`: 正常系と失敗系の回帰 fixture

## 5. Done の定義

次の状態をもって「実装準備完了」とみなす。

- 要件定義と仕様書に矛盾がない
- JSON Schema で request / response / error を検証できる
- 正常系と失敗系のサンプルが揃っている
- CLI / HTTP API / MCP が同一契約を参照する前提がドキュメント化されている
- 実装順、受け入れ条件、テスト観点が別ファイルとして整理されている

## 6. 将来拡張の境界

今回の仕様は MVP を対象にする。以下は将来拡張として扱う。

- 非同期実行と run store
- 複数課題バッチ入力
- 外部システム連携
- 推薦ロジックの自動最適化
- 長期履歴の学習や継続的改善
